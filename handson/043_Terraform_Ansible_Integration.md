## Topic

Terraform and Ansible are complementary tools. They solve different parts of the infrastructure lifecycle:

| Tool | Responsibility | State | Language |
|---|---|---|---|
| Terraform | Provision infrastructure (EC2, VPC, RDS, S3) | Stateful (terraform.tfstate) | Declarative HCL |
| Ansible | Configure software on servers (nginx, app deploys, users) | Stateless (idempotent tasks) | Declarative YAML |

The integration pattern is simple:

```
terraform apply
    │
    └── creates EC2 instance
            │
            └── local-exec provisioner fires
                    │
                    └── ansible-playbook -i '<ip>,' playbooks/install_nginx.yml
                                │
                                └── installs nginx, deploys robochef.co config
```

Terraform handles the "what exists" question. Ansible handles the "what runs on it" question. Neither needs to know the other's internals.

## Step 4 — playbooks/install_nginx.yml

This is the Ansible playbook that Terraform will call after the EC2 instance is ready. It installs nginx, writes a virtual host config for robochef.co, enables the site, and restarts nginx.

```yaml
# playbooks/install_nginx.yml
---
- hosts: all
  become: true
  vars:
    app_name: robochef
    app_domain: robochef.co

  tasks:
    - name: Update apt package cache
      apt:
        update_cache: yes
        cache_valid_time: 3600

    - name: Install nginx
      apt:
        name: nginx
        state: present

    - name: Ensure nginx is started and enabled
      service:
        name: nginx
        state: started
        enabled: yes

    - name: Remove default nginx site
      file:
        path: /etc/nginx/sites-enabled/default
        state: absent
      notify: Restart nginx

    - name: Deploy robochef.co nginx virtual host config
      copy:
        content: |
          # Generated by Terraform + Ansible
          server {
              listen 80;
              server_name {{ app_domain }};

              location / {
                  return 200 "Hello from {{ app_domain }} — served by nginx on $(hostname)!";
                  add_header Content-Type text/plain;
              }

              location /health {
                  return 200 "OK";
                  add_header Content-Type text/plain;
              }
          }
        dest: /etc/nginx/sites-available/{{ app_name }}
        owner: root
        group: root
        mode: "0644"
      notify: Restart nginx

    - name: Enable the robochef.co nginx site
      file:
        src: /etc/nginx/sites-available/{{ app_name }}
        dest: /etc/nginx/sites-enabled/{{ app_name }}
        state: link
      notify: Restart nginx

    - name: Verify nginx config syntax
      command: nginx -t
      changed_when: false

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted
```

Key Ansible concepts used here:

| Concept | Explanation |
|---|---|
| `become: true` | Run tasks with sudo (needed for apt, systemctl, /etc writes) |
| `notify: Restart nginx` | Trigger the handler only when the task reports `changed` |
| Handlers | Run once at the end of the play, even if notified multiple times |
| `copy` module | Write literal content to a file on the remote server |
| `file` module with `state: link` | Create a symlink (nginx sites-enabled pattern) |
| `changed_when: false` | `nginx -t` is a read-only check; never mark it as changed |

## Step 5 — main.tf

```hcl
# main.tf

# ── SSH Key Pair ────────────────────────────────────────────────────────────

resource "tls_private_key" "demo" {
  algorithm = "RSA"
  rsa_bits  = 4096
}

resource "local_sensitive_file" "private_key" {
  filename        = "/tmp/terraform-043-demo.pem"
  content         = tls_private_key.demo.private_key_pem
  file_permission = "0600"
}

resource "aws_key_pair" "demo" {
  key_name   = "terraform-043-ansible-demo"
  public_key = tls_private_key.demo.public_key_openssh
}

# ── Security Group ──────────────────────────────────────────────────────────

resource "aws_security_group" "web" {
  name        = "terraform-043-web-sg"
  description = "Allow SSH and HTTP for Ansible integration demo"

  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]  # restrict to your IP in production
  }

  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name    = "terraform-043-web-sg"
    Project = "terraform-043"
  }
}

# ── EC2 Instance ────────────────────────────────────────────────────────────

resource "aws_instance" "web" {
  ami                         = var.ami_id
  instance_type               = var.instance_type
  key_name                    = aws_key_pair.demo.key_name
  vpc_security_group_ids      = [aws_security_group.web.id]
  associate_public_ip_address = true

  tags = {
    Name        = "${var.app_name}-ansible-demo"
    Project     = "terraform-043"
    ManagedBy   = "terraform+ansible"
    Application = var.app_name
  }
}

# ── Ansible Provisioner ─────────────────────────────────────────────────────
#
# This null_resource fires after the EC2 instance is created.
# It calls ansible-playbook on the local machine via local-exec.
# The instance IP is passed as an inline inventory (-i '<ip>,').
# The trailing comma after the IP tells Ansible this is an inline
# inventory string, not a file path.

resource "null_resource" "ansible_provision" {
  depends_on = [aws_instance.web, local_sensitive_file.private_key]

  # Re-run Ansible if the instance is replaced OR the playbook changes
  triggers = {
    instance_id   = aws_instance.web.id
    playbook_hash = filemd5("${path.module}/playbooks/install_nginx.yml")
  }

  provisioner "local-exec" {
    command = <<-EOC
      echo "Waiting 30 seconds for SSH to become available on ${aws_instance.web.public_ip}..."
      sleep 30

      echo "Running Ansible playbook against ${aws_instance.web.public_ip}..."
      ANSIBLE_HOST_KEY_CHECKING=False \
      ANSIBLE_SSH_RETRIES=3 \
      ansible-playbook \
        -i '${aws_instance.web.public_ip},' \
        -u ubuntu \
        --private-key ${local_sensitive_file.private_key.filename} \
        --extra-vars "app_name=${var.app_name} app_domain=${var.app_domain}" \
        playbooks/install_nginx.yml

      echo "Ansible provisioning complete."
    EOC
  }
}
```

### Why `sleep 30`?

When an EC2 instance is created, it takes time for:
1. The hypervisor to start the VM
2. The OS to boot
3. The `sshd` daemon to start accepting connections

Terraform marks the `aws_instance` resource as "created" as soon as the AWS API confirms the instance exists — not when SSH is ready. The `sleep 30` gives the instance time to reach SSH readiness. For slower instance types or heavily loaded regions, increase this to `sleep 60`.

A more robust approach uses the `aws_instance` `user_data` to signal readiness, or uses Ansible's `--timeout` and `ANSIBLE_SSH_RETRIES` to retry automatically.

### Inline inventory `-i '<ip>,'`

```bash
ansible-playbook -i '192.0.2.10,' ...
```

The trailing comma after the IP address is required. It tells Ansible that the value is an inline inventory string (a comma-separated list of hosts), not a path to an inventory file. Without the comma, Ansible tries to open `192.0.2.10` as a file and fails.

### `ANSIBLE_HOST_KEY_CHECKING=False`

The first SSH connection to a new server normally prompts:
```
The authenticity of host '...' can't be established. Are you sure you want to continue connecting?
```
Setting `ANSIBLE_HOST_KEY_CHECKING=False` disables this check so the playbook can run non-interactively. In production, manage known_hosts properly or use AWS Systems Manager Session Manager instead of direct SSH.

## Step 9 — Re-running Ansible After a Playbook Change

The `triggers` map in `null_resource.ansible_provision` includes:

```hcl
triggers = {
  instance_id   = aws_instance.web.id
  playbook_hash = filemd5("${path.module}/playbooks/install_nginx.yml")
}
```

`filemd5()` returns the MD5 hash of the playbook file. If you edit `install_nginx.yml`, the hash changes, Terraform detects the trigger change, and re-runs the `null_resource` — which re-runs Ansible. This is the correct way to force re-provisioning without destroying the EC2 instance.

Try it:
1. Edit `playbooks/install_nginx.yml` — change the `return 200` message.
2. Run `terraform apply -auto-approve`.
3. Watch Ansible re-run against the existing instance.
4. Curl again to see the new message.

## Step 10 — Dynamic Inventory (Production Pattern)

In the demo, we pass the IP inline (`-i '<ip>,'`). In a real project with multiple servers, you generate an Ansible inventory file instead:

```hcl
# Generate an inventory file from all managed instances
resource "local_file" "ansible_inventory" {
  filename = "/tmp/ansible-inventory-043.ini"
  content  = <<-INI
    [webservers]
    ${aws_instance.web.public_ip} ansible_user=ubuntu ansible_ssh_private_key_file=${local_sensitive_file.private_key.filename}

    [webservers:vars]
    ansible_host_key_checking=False
  INI
}

resource "null_resource" "ansible_with_inventory" {
  depends_on = [local_file.ansible_inventory]

  triggers = {
    instance_id   = aws_instance.web.id
    playbook_hash = filemd5("${path.module}/playbooks/install_nginx.yml")
  }

  provisioner "local-exec" {
    command = <<-EOC
      sleep 30
      ANSIBLE_HOST_KEY_CHECKING=False ansible-playbook \
        -i ${local_file.ansible_inventory.filename} \
        playbooks/install_nginx.yml
    EOC
  }
}
```

For large fleets, use the [aws_ec2 dynamic inventory plugin](https://docs.ansible.com/ansible/latest/collections/amazon/aws/aws_ec2_inventory.html) which queries the AWS API directly and groups instances by tags, region, or VPC.

## Integration Patterns — Decision Guide

| Pattern | When to use |
|---|---|
| Terraform → local-exec → ansible-playbook | Simple projects; single-account; small teams |
| Terraform → user_data (cloud-init) | Package install + simple config at first boot only |
| Terraform → Packer → golden AMI | Compliance-heavy environments; fast instance boot |
| Terraform Cloud + Ansible Tower / AWX | Enterprise; audit trail; RBAC; multi-team |
| Terraform → SSM Run Command | No public IP needed; AWS-only environments |

## Separation of Concerns — Why This Matters

```
┌───────────────────────────────────────────────────────────┐
│  TERRAFORM                                                │
│  ─────────                                                │
│  EC2 instance exists?         ← Terraform manages this   │
│  Security group rules correct? ← Terraform manages this  │
│  RDS database provisioned?    ← Terraform manages this   │
└───────────────────────────────────────────────────────────┘
                        │
                        │ local-exec hands off
                        ▼
┌───────────────────────────────────────────────────────────┐
│  ANSIBLE                                                  │
│  ───────                                                  │
│  nginx installed?             ← Ansible manages this     │
│  App config deployed?         ← Ansible manages this     │
│  Users and permissions set?   ← Ansible manages this     │
└───────────────────────────────────────────────────────────┘
```

Neither tool duplicates the other's work. Terraform does not know about nginx. Ansible does not know about EC2 resource IDs. The only coupling point is the IP address passed from Terraform to Ansible via the local-exec command.

## Key Concepts Summary

| Concept | Detail |
|---|---|
| local-exec bridge | Terraform calls `ansible-playbook` via `local-exec` after creation |
| Inline inventory | `-i '<ip>,'` — trailing comma is required |
| `ANSIBLE_HOST_KEY_CHECKING=False` | Allows non-interactive SSH to new hosts |
| `sleep 30` | Gives the EC2 instance time to reach SSH readiness |
| `triggers.playbook_hash` | Re-runs Ansible when the playbook file changes |
| `filemd5()` | Terraform built-in function to hash a file's contents |
| Handlers | Ansible restarts nginx only when a notifying task changes |
| `become: true` | Ansible uses sudo for privileged operations |
| Separation of concerns | Terraform = infrastructure, Ansible = configuration |
