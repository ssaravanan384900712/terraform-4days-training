## Topic

`templatefile(path, vars)` reads a file from disk and substitutes template variables and directives before returning the rendered string. It is the recommended way to inject dynamic values into:

- **Nginx / Apache configs** — domain, port, upstream app addresses
- **cloud-init / user_data scripts** — instance-specific settings at boot time
- **JSON / YAML config files** — environment-specific service configuration
- **Shell scripts** — injecting secrets, bucket names, or endpoints at plan time

The two things `templatefile()` adds on top of a plain `file()` read:

| Function | Reads file | Substitutes `${var}` | Supports `%{ for }` / `%{ if }` |
|---|---|---|---|
| `file(path)` | yes | no | no |
| `templatefile(path, vars)` | yes | yes | yes |

---

## Template Directives Quick Reference

```
${variable}                    — interpolate a variable
${object.attribute}            — interpolate an attribute

%{ if condition ~}             — conditional block (~ strips trailing newline)
%{ else ~}
%{ endif ~}

%{ for item in list ~}         — loop block
%{ endfor ~}
```

The trailing `~` on a directive strips the newline that would otherwise appear after the directive line, keeping the output clean.

## Step 3 — templates/site.conf.tpl

This is the template file. It uses both `${variable}` interpolation and the `%{ for ... ~}` loop directive.

```
# Site configuration generated by Terraform
# Owner: ${owner}
# Generated: ${generated_at}

server {
    server_name ${domain};
    listen      ${port};
    root        /var/www/${app_name};

    location / {
        proxy_pass http://127.0.0.1:${app_port};
    }

    # Enabled features:
%{ for feature in features ~}
    # - ${feature}
%{ endfor ~}
}
```

Save this file at `templates/site.conf.tpl`.

What each placeholder does:

| Placeholder | Type | Description |
|---|---|---|
| `${owner}` | string | Person or team responsible for this site |
| `${generated_at}` | string | Timestamp stamped at plan/apply time |
| `${domain}` | string | Virtual host name |
| `${port}` | number | Port nginx listens on (80 or 443) |
| `${app_name}` | string | Web root folder name under /var/www |
| `${app_port}` | number | Upstream application port |
| `features` | list(string) | Feature flags to document in the config |

## Step 4 — main.tf

```hcl
# main.tf

# ── Rendered configs live in locals ────────────────────────────────────────
locals {
  robochef_config = templatefile("${path.module}/templates/site.conf.tpl", {
    owner        = "saravanans"
    domain       = "robochef.co"
    port         = 443
    app_name     = "robochef"
    app_port     = 3000
    generated_at = timestamp()
    features     = ["auth", "analytics", "dark-mode"]
  })

  chillbot_config = templatefile("${path.module}/templates/site.conf.tpl", {
    owner        = "saravanans"
    domain       = "chillbotindia.com"
    port         = 80
    app_name     = "chillbot"
    app_port     = 8080
    generated_at = timestamp()
    features     = ["chat", "notifications"]
  })
}

# ── Write the rendered configs to /tmp so you can cat them ─────────────────
resource "local_file" "robochef_nginx" {
  filename = "/tmp/robochef-nginx.conf"
  content  = local.robochef_config
}

resource "local_file" "chillbot_nginx" {
  filename = "/tmp/chillbot-nginx.conf"
  content  = local.chillbot_config
}
```

Key points:

- `templatefile()` is called inside `locals {}` so the rendered strings are computed once and reused.
- `path.module` resolves to the directory that contains `main.tf`, so the relative path to the template is always correct regardless of where `terraform` is invoked.
- `timestamp()` is a built-in Terraform function that returns the current UTC time as an RFC 3339 string. It is re-evaluated every `terraform apply`, which means Terraform will detect a drift on repeated applies and re-write the file. This is expected and harmless for config files.

## Step 8 — file() vs templatefile() Side-by-Side

To understand the difference, add this experiment to your main.tf temporarily:

```hcl
# main.tf — add this at the bottom for the demo
locals {
  # file() — reads raw bytes, returns them unchanged
  raw_template = file("${path.module}/templates/site.conf.tpl")

  # templatefile() — reads and substitutes
  rendered = templatefile("${path.module}/templates/site.conf.tpl", {
    owner        = "demo"
    domain       = "example.com"
    port         = 80
    app_name     = "demo-app"
    app_port     = 3000
    generated_at = "2026-05-21"
    features     = ["feature-x"]
  })
}

output "raw_vs_rendered" {
  value = {
    raw      = local.raw_template   # still has ${domain}, ${port}, etc.
    rendered = local.rendered       # substituted values
  }
}
```

Run `terraform apply -auto-approve` and compare the two outputs. The raw string still contains `${domain}`, `${port}`, and the `%{ for ... }` directives as literal text. The rendered string has them replaced.

## Step 9 — Conditional Directives

Add an `if` block to the template to show conditional sections:

```
# templates/site-advanced.conf.tpl

server {
    server_name ${domain};
    listen      ${port};

%{ if port == 443 ~}
    ssl_certificate     /etc/ssl/certs/${domain}.crt;
    ssl_certificate_key /etc/ssl/private/${domain}.key;
%{ endif ~}

    location / {
        proxy_pass http://127.0.0.1:${app_port};
    }
}
```

Reference it in main.tf:

```hcl
locals {
  robochef_ssl_config = templatefile("${path.module}/templates/site-advanced.conf.tpl", {
    domain   = "robochef.co"
    port     = 443
    app_port = 3000
  })

  chillbot_plain_config = templatefile("${path.module}/templates/site-advanced.conf.tpl", {
    domain   = "chillbotindia.com"
    port     = 80
    app_port = 8080
  })
}
```

The `robochef_ssl_config` will include the `ssl_certificate` lines because `port == 443`. The `chillbot_plain_config` will skip them because `port == 80`.

## Key Concepts Summary

| Concept | Detail |
|---|---|
| `templatefile(path, vars)` | Reads a file and substitutes vars; returns the rendered string |
| `${var}` | Standard HCL interpolation inside template files |
| `%{ for item in list ~}` | Loop directive; `~` strips the trailing newline |
| `%{ if condition ~}` | Conditional directive |
| `path.module` | Built-in reference to the current module's directory |
| `timestamp()` | Returns current UTC time; causes re-apply drift — acceptable for generated configs |
| `file()` | Reads file as-is; no substitution |

## Common Errors

**Error: Invalid template variable type**
```
Call to function "templatefile" failed: vars map does not allow null values.
```
Fix: make sure every variable referenced in the template is present in the vars map. A missing key causes this error.

**Error: Extra `%{ endfor }` without matching `%{ for }`**
```
Error in template: Unexpected end of template directive
```
Fix: each `%{ for ... ~}` must have a matching `%{ endfor ~}`.

**Blank lines in loop output**
If you see extra blank lines between loop iterations, add the `~` strip marker: `%{ for item in list ~}` and `%{ endfor ~}`.
